# 一个类如果什么都不写，编译器会在需要时（on-demand）为它隐式生成以下几个特殊成员函数（special member functions）：

| 特殊成员函数                            | 功能   | 隐式生成条件（简化说明）                            |
| --------------------------------- | ---- | --------------------------------------- |
| **默认构造函数** `T()`                  | 构造对象 | 如果你没有写任何构造函数，编译器才会生成                    |
| **析构函数** `~T()`                   | 销毁对象 | 总是会生成，除非你自己写了                           |
| **复制构造函数** `T(const T&)`          | 拷贝构造 | 如果需要拷贝且你没有写禁用/自定义                       |
| **复制赋值运算符** `operator=(const T&)` | 拷贝赋值 | 同上                                      |
| **移动构造函数** `T(T&&)`               | 移动构造 | C++11 起，如果没有用户定义的拷贝操作（复制构造、复制赋值），且成员可移动 |
| **移动赋值运算符** `operator=(T&&)`      | 移动赋值 | 同上                                      |

一旦写了构造函数，默认构造不再自动生成

一旦写了析构函数，默认移动构造/赋值就不再自动生成

一旦写了复制构造/赋值，默认移动构造/赋值就不再自动生成

遵循原则：Rule of Zero，Rule of Five，Rule of Delete
- 如果类不管理资源(没有new,delete,socket,fd等)，就什么都不写，让编译器自动生成
- 当内部有资源（内存、句柄、文件、互斥锁等），就全部写出来
- 如果类要求不能被复制或者移动，写出来delete

# 动态多态靠虚函数实现

虚函数是通过对象的动态类型决定调用目标的成员函数，是多态的体现。一般编译器实现都是通过虚表实现，但虚表并不是C++标准。

虚函数实际调用路径是在调用时，先通过虚函数指针找到找到虚函数表，在虚函数表内找到实际函数地址后call

每个**有虚函数的子对象**都有一个vptr，如果有继承，则可能有多个

多继承时，则有多套虚函数表和虚表指针，一般子类的base指针为第一个父类的，第二个父类的this指针需要被thunk。例如类C继承类A和类B，分别都有各自的虚函数f1,f2,f3，则最后C只有两个虚指针和两个虚表，类C的f3实际地址被放到了类A的虚表中

菱形继承时取决于是否虚继承，虚继承则只有一份A，否则有两份A。

构造函数不能是虚函数，因为虚函数调用需要 vptr，而 vptr 在构造函数执行过程中才初始化。
此外，对象在构造阶段并不具备完整的动态类型特征，因此构造过程不能发生多态。编译器也会报错

可以在构造/析构函数中调用虚函数，但是是静态绑定的，不会多态。因为子对象可能还不存在，调用子类虚函数会访问未初始化数据，安全原因

凡是打算通过基类指针delete的类，析构函数必须是virtual，否则只会析构父类，而子类资源会泄露

重载是依靠编译期名字查找和重载决议的规则进行的，要先找到名字，再重载决议，因此会有继承父类构造函数，否则子类找不到构造函数名字，还谈不上重载决议

虚表在GCC里前两项一般是RTTI相关的内容，第三项开始才是虚函数指针

第一项是offset-to-top表示从当前子对象到最顶层对象的偏移，主要用于多继承下的dynamic_cast。单继承通常永远是0

第二项是type_info*指向type_info结构体，与RTTI相关。
type_info结构体内容在编译期写入.rodata段，dynamic_cast时通过查type_info(Class)里的所有相关类型指针，判断是否存在父类，是否public，是否唯一等，计算this偏移后返回实际指针。没找到则返回nullptr。

# cast类型：C风格转换等于无视类型系统，C++不要使用

| cast 类型              | 安全性       | 常用途          | 运行期检查 | 典型危险点               |
| -------------------- | --------- | ------------ | ----- | ------------------- |
| **static_cast**      | 较安全       | 基本类型转换、向上转型  | ❌     | 错误 downcast 会 UB    |
| **dynamic_cast**     | 最安全（多态场景） | 向下转型、运行期类型检查 | ✔     | 性能开销、要求 RTTI        |
| **const_cast**       | 危险        | 去掉 const     | ❌     | 修改真正的 const 对象 → UB |
| **reinterpret_cast** | 最危险       | 底层位级转换       | ❌     | 易破坏类型系统             |

dynamic_cast工作原理：

1. 类型信息查询：通过对象的虚函数表找到其RTTI信息
2. 类型层次遍历：检查目标类型是否在对象的继承层次中
3. 指针调整：对于多重继承，调整this指针到正确的子对象位置
4. 返回结果：如果转换合法返回正确指针，否则返回nullptr（对于指针）或抛出bad_cast（对于引用）

# final能提高代码执行效率？

只在有虚函数和动态分派的时候可能优化，被final修饰后，可能可以直接调用函数而不用走虚函数表，但性能有限，因此final允许优化，而不是保证优化
