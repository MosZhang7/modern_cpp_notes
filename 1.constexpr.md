# static const constexpr/consteval/constinit

| 关键字           | 可修饰全局变量 | 可修饰局部变量 | 可修饰成员变量 | 可修饰函数       | 可修饰成员函数 | 可修饰类       | 说明                      |
| ------------- | ------- | ------- | ------- | ----------- | ------- | ---------- | ----------------------- |
| **static**    | ✔       | ✔       | ✔       | ✔           | ✔       | ✔          | 最多用途的关键字                |
| **const**     | ✔       | ✔       | ✔       | ✔（顶层 const） | ✔       | ❌          | “只读语义”, 但不能修饰类          |
| **constexpr** | ✔       | ✔       | ✔       | ✔           | ✔       | ✔（C++20 起） | 编译期能力                   |
| **consteval** | ❌       | ❌       | ❌       | ✔（普通函数）     | ✔       | ✔（C++20 起） | 强制编译期执行                 |
| **constinit** | ✔       | ❌       | ✔（静态成员） | ❌           | ❌       | ❌          | 仅能修饰 static-duration 对象 |


## static:控制生命周期 & 链接性(仅限本TU可见),不是常量相关关键字。

修饰全局变量：static int x = 5;影响为仅限本TU可见，因为全局变量本身就具备static语义，放在.data或者.bss

修饰局部变量：void f() {
    static int counter = 0;
};可以延长生命周期至程序结束，多次调用函数f()全局只会初始化一次counter，因此多次对counter++，结果会变，而不总是1。在C++11后，static修饰的局部变量的初始化是线程安全的(仅仅是初始化，后面的读写仍然可能竞争)。

修饰成员变量：struct A {
    static int n;
};所有A的实例共享一份n，属于类本身，不属于任何一个实例，因此需要在类外定义一次：int A::n = 0;
需要注意是属于类本身，因此模板类里如果有static变量，很可能会因为实例化了多个类，而有多份static变量

修饰全局函数：影响为仅限本TU可见

修饰成员函数：可以作为普通函数调用，不依赖对象this指针

修饰类：static class x{} 无任何实际作用，不会变成静态类，影响为仅限本TU可见


## const:关心只读语义

修饰全局/局部/成员变量：const int x = 10;初始化后不可再更改

修饰成员函数：int get() const;表示该函数不修改任何非mutable成员变量。const的类的实例只能调用const成员函数。

## constexpr/eval/init:关心是否编译期处理

constexpr和consteval自带const语义，constinit则只关心编译期初始化，与const无关

constexpr修饰全局/局部/成员变量：变量必须是一个constant expression，能够进行编译期求值，如果不是，编译器会直接报错，不能退化。
constexpr修饰函数：有资格成为编译期函数，是否编译期求值，取决于调用处的参数是否是constant expression。不满足时可以退化为普通函数。

consteval修饰函数:比constexpr更严格，不可退化。只能修饰函数，是因为变量用constexpr也不可退化。

constinit只能修饰static storage duration变量：要求在编译期初始化，不具备const语义，解决静态变量的动态初始化的问题

constexpr和consteval修饰类/构造函数/析构函数等，都是用于大量编译期执行的用法

经常可以用于：
1. if constexpr做模板的编译期分支优化，毕竟可以直接删掉另一条路的代码
2. 提前计算复杂的数，例如fib()
3. 编译期构建lookup Table，直接构建出表，或者构建特殊字符串等

# structured binding C++17

有点类似于语法糖，auto返回内部类型，pair或者tuple，通过.get获取值

结构化绑定只能绑定三种类型
1. 数组:
``` C++
int a[3] = {1,2,3};
auto [x, y, z] = a;
```
2. tuple-like: std::pair, std::tuple, std::array，以及ranges::views::zip后的内容
3. Aggregate聚合类型

聚合类型在 C++17 的定义：
- 没有任何自定义构造函数
- 没有 private/protected 的非静态成员（成员都得 public）
- 没有虚函数
- 没有虚基类
例如
``` C++
struct A {
    int x;
    double y;
};           // ✔ 是 aggregate → 可以结构化绑定
```

最常用于for循环内解构auto对象的item，例如vector，map等，例如：
``` C++
std::map<std::string, int> m = {{"A", 1}, {"B", 2}, {"C", 3}};

for (auto&& [key, value] : m) {
    value += 10;
}
```

## std::tie

C++11和std::tuple一起出现，主要用于解tuple的包，用于生成一个tuple里数据的引用视图，不拷贝不拥有数据。
在C++17有了结构化绑定之后就不再用于解包

现在则主要用于多字段以不同key进行比较或排序。因为只是引用tuple-like里的元素，因此开销很小，可以合包作为视图，且tuple天然严格弱序
主要用于comparator，而不是projection，projection更推荐值语义

# 类型推导

## auto：使用模板参数推导规则，推导出干净的值类型，不含const不含引用

auto只给出最原始的值类型，不管类别等，需要自己加
例如：auto&和auto &&


## decltype：推导出表达式的类型，会包括const和引用类型等

表达式推导规则：

| 表达式值类别  | 返回类型      |
| ------- | --------- |
| lvalue  | `T&`      |
| xvalue  | `T&&`     |
| prvalue | `T` （对象值） |

当仅仅传入变量名而非表达式时，decltype直接返回变量声明的类型(不是auto的纯值类型)以方便。
而加括号之后会成为表达式，会使用表达式推导方式，因此需要小心使用括号，例如
``` C++
int x = 0;

decltype(auto) c1 = (x); // int&   走表达式推导方式
decltype(auto) c2 = x;   // int    走变量推导方式
decltype(auto) c3 = std::move(x) // int&&

// decltype(auto)：走的是decltype的推导方式，auto仅仅做占位符，将表达式挪过来而已
```

# 一个类如果什么都不写，编译器会在需要时（on-demand）为它隐式生成以下几个特殊成员函数（special member functions）：

| 特殊成员函数                            | 功能   | 隐式生成条件（简化说明）                            |
| --------------------------------- | ---- | --------------------------------------- |
| **默认构造函数** `T()`                  | 构造对象 | 如果你没有写任何构造函数，编译器才会生成                    |
| **析构函数** `~T()`                   | 销毁对象 | 总是会生成，除非你自己写了                           |
| **复制构造函数** `T(const T&)`          | 拷贝构造 | 如果需要拷贝且你没有写禁用/自定义                       |
| **复制赋值运算符** `operator=(const T&)` | 拷贝赋值 | 同上                                      |
| **移动构造函数** `T(T&&)`               | 移动构造 | C++11 起，如果没有用户定义的拷贝操作（复制构造、复制赋值），且成员可移动 |
| **移动赋值运算符** `operator=(T&&)`      | 移动赋值 | 同上                                      |

一旦写了构造函数，默认构造不再自动生成

一旦写了析构函数，默认移动构造/赋值就不再自动生成

一旦写了复制构造/赋值，默认移动构造/赋值就不再自动生成

遵循原则：Rule of Zero，Rule of Five，Rule of Delete
- 如果类不管理资源(没有new,delete,socket,fd等)，就什么都不写，让编译器自动生成
- 当内部有资源（内存、句柄、文件、互斥锁等），就全部写出来
- 如果类要求不能被复制或者移动，写出来delete

# cast类型：C风格转换等于无视类型系统，C++不要使用

| cast 类型              | 安全性       | 常用途          | 运行期检查 | 典型危险点               |
| -------------------- | --------- | ------------ | ----- | ------------------- |
| **static_cast**      | 较安全       | 基本类型转换、向上转型  | ❌     | 错误 downcast 会 UB    |
| **dynamic_cast**     | 最安全（多态场景） | 向下转型、运行期类型检查 | ✔     | 性能开销、要求 RTTI        |
| **const_cast**       | 危险        | 去掉 const     | ❌     | 修改真正的 const 对象 → UB |
| **reinterpret_cast** | 最危险       | 底层位级转换       | ❌     | 易破坏类型系统             |

dynamic_cast工作原理：

1. 类型信息查询：通过对象的虚函数表找到其RTTI信息
2. 类型层次遍历：检查目标类型是否在对象的继承层次中
3. 指针调整：对于多重继承，调整this指针到正确的子对象位置
4. 返回结果：如果转换合法返回正确指针，否则返回nullptr（对于指针）或抛出bad_cast（对于引用）
