# shared_ptr

内部主要有两个指针，一个指向被管理对象T*，另一个指向控制块control_block*(共享)。shared_ptr共享的是控制块，而不是对象本身

控制块里一般存放的是：
control_block
 ├── atomic<long> strong_count  // shared_ptr 数量
 ├── atomic<long> weak_count    // weak_ptr 数量 + **strong_count**
 ├── deleter                    // 删除策略
 └── allocator (可选)

strong_count == 0 对象被析构，但控制块不一定释放，要等weak_count == 0才会析构控制块

不建议用裸指针构造，因为可能构造多个shared_ptr造成最后双重释放，且先分配T再分配control_block，两次分配。

建议make_shared，一次分配完毕。但weak_ptr没释放完毕时，整块内存都不会被释放

引用计数是atomic类型，因此引用计数是线程安全的，但针对对象的操作不是线程安全的。

常用方法：
- reset(p):放弃当前对象，再接管 p，也可以不传参，直接放弃管理，使得强引用数量--
- swap:交换两个的对象指针和控制块指针
- unique():返回bool,当前状态强引用计数是否为1
- use_count()

allocate_shared基本与make_shared一样，但第一个参数可以传入内存区域，控制后面的东西放到指定内存，一般用于内存池

循环引用需要依靠weak_ptr解决

## aliasing constructor

template<class Y>
shared_ptr(const shared_ptr<Y>& r, T* ptr) noexcept;

r提供control block
ptr提供普通指针

``` C++
auto pa = std::make_shared<int>(42);

int* pb = new int(100);

std::shared_ptr<int> spb(pa, pb);  // aliasing 构造
将pa生命周期延长，直到spb析构，pa才会被析构.因为只是绑定生命周期，不会更改deleter，因此pb则需要手动析构，否则会内存泄漏
```

## enable_share_from_this

直接return this指针会生成一个新的控制块，最后本对象会被析构两次导致崩溃。

shared_ptr 构造过程中，检测到 Foo 继承了 enable_shared_from_this<Foo>， 就把“自己的控制块”写进 Foo::weak_this`。

``` C++
struct Foo : enable_shared_from_this<Foo> {
  shared_ptr<Foo> get_self() {
    return weak_from_this();
    return shared_from_this();
  }
};
```

对象必须已经由 shared_ptr 管理，否则UB，因此不可在构造函数里调用，因为构造期间，shared_ptr还没完成控制块绑定，weak_this还没被设置
