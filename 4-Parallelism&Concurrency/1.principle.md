# 并行与并发

并行描述的是一种物理能力，是多个任务在不同CPU核心上真正同时执行，需要有硬件支持多核多线程才可能并行

并发描述的是一种结构，所有任务在一段时间内被轮流推进，因此只有一个CPU核心也是可以并发的

并行关心如何把任务交给更多的CPU去运行，而并发则关心多个任务间如何安全地交互一起推进

两个线程同时启动，是否并行，不取决于有没有发生竞争，而看两个任务是否真的在两个CPU上运行。竞争资源只会影响并行效率，不会决定物理上是否真的在并行。即使暂时发生竞争导致其中一个被阻塞，也是并行，只是属于并行退化，暂时无法有效利用并行性

# 为什么一定要有std::mutex或std::atomic才能避免data race？

因为指令重排。为了更高的效率，现代CPU和编译器会发生指令重排和乱序执行等，将多次读/写合并成一次，导致看起来顺序执行的代码，实际上并不一定。

A线程修改变量，B线程可能无法及时看到等，因此数据竞争的本质是**可见性 和 顺序性**被破坏

因此mutex和atomic是用于阻止CPU和编译器对关键指令的重排，保证不同线程对内存的可见性，以及读写操作的原子性

普通变量不能用于多线程，是因为编译器以及CPU视角来看，会将普通变量认定为只有单线程视角在操作，因此为了效率，会在单线程视角下不改变语义的情况下，进行无限制的优化。

要告诉编译器某些变量是多线程在使用，不要随意优化的方式，就是使用mutex和atomic

只要变量可能被多个线程同时访问且没有同步，则整个程序都是UB。因为有时候编译器甚至可以直接优化掉整个thread

原子性，顺序性，可见性保证了C++的同步语义

原子性：某个操作不可被分割
顺序性：指令的先后顺序
可见性：写入后在何时能让其他线程看见

# Happens-Before

HB模型的构成：sequenced-before + synchronizes-with + transitivity
sequenced-before：单线程的顺序性
synchronizes-with：跨线程的同步性
transitivity：结论的可传递性

最常用的HB：C++规定了原子操作里配对的store-release和load-acquire构成HB

还有一些场景：
- mutex.unlock和mutex.lock，代表release和acquire
- condition_variable 的 notify HB wait，因为wait会先mutex.unlock直到被notify会尝试mutex.lock
- std::thread::join，线程中所有操作 HB join()返回，本身是语言设计的强同步，不依靠atomic或mutex
- future / promise：promise.set_value HB future.get，内部用原子 / 条件变量实现

store-release和load-acquire构成的屏障是影响该HB前/后的操作，而非单个变量

# 条件变量cv和原子条件变量

条件变量wait后会释放锁，并等待被唤醒，因此一定要拿到锁才可以wait。所以基本与unique_lock绑定，因为可以拿到锁且可以多次手动解锁上锁

唤醒前会拿到锁检查条件，条件满足则真正唤醒开始运行，否则继续wait。因此可能需要多次wait否则永远醒不来

正确使用条件变量一定要带pred，防止假唤醒。如果wait时条件满足，则立即返回，不会休眠。
假唤醒是硬件层、操作系统层面的允许，为了性能，操作系统会主动地多唤醒

正常来说，无脑使用notify_all即可，因为有假唤醒的存在，所以实际性能差别不大
