# std::thread

将callable丢给它，可以在另一个执行流中执行。一创建就启动。析构时必须要保证join()或detach()。可以判断joinable()，否则std::terminate()
在C++20可以被jthread替代。

std::thread由于运行时需要保证参数的生命周期，因此内部会按值保存所有参数，在实际运行调用函数时，会将内部参数std::move()后传给函数。因此内部保存的值类型，不能传给形参为左值引用的参数，因为会std::move变为右值引用，除非用std::ref，用了之后，invoke会针对被move后的ref进行特判，传递引用

# std::future / std::promise

std::future 与 std::promise 用于把线程间通信 提升为 值/异常的异步传递。C++规定异常不能跨线程传播，线程中有异常没被捕获，则直接整个进程terminate。而future / promise机制可以将异常保存下来，到同一个线程再rethrow

创建promise时会创建一套共享内存区shared state，结构如下：其中promise和future都只是handle，future要先有shared state才可以
        ┌──────────────────────────┐
        │        shared state      │
        │                          │
        │  - value OR exception    │
        │  - ready flag            │
        │  - synchronization       │
        └──────────────────────────┘
             ▲                 ▲
             │                 │
        promise<T>         future<T>
      (写端 / producer)   (读端 / consumer)

由于future和promise都是1对1，因此不能复制，只能移动。

# promise<T>语义：未来一定会给你一个T，或者一个异常

promise析构前需要已经set过，否则会往shared state里存入broken_promise异常。

promise只有三个API
``` C++
// 把 T / exception 存进 shared state
// 标记ready = true
// 唤醒所有等待这个state的线程
// 建立HB：set_value HB future.get()
// value和exception 只能set一次，future.get获得第一次的结果。第二次set会在本线程抛异常。需要在本线程捕获，否则整个进程崩溃
promise.set_value(T); 
promise.set_exception(std::exception_ptr);

// 从shared state生成一个观察句柄，即为future<T>，只能调用一次
auto f = promise.get_future();

// 防止普通的set一设置，立刻唤醒了等待线程。基本不用
promise.set_xxx_at_thread_exit(x);
```

# future<T>语义：对shared state中内容的唯一消费者
``` C++

// get是无条件阻塞，直到 shared state = ready
// 销毁性读取，get后放弃对shared state的观察，即无法进行第二次get
// 返回类型为 T，表示move语义
future.get(std::exception_ptr);

// 将唯一消费者语义转为多观察者语义，自身不再观察shared state，不再可用
// 每个f_s都可以copy，move等，可以多次get，但返回的是const T&类型，表示只读语义，没有copy和move语义
shared_future<T> f_s = future.share();

// 相当于get的前半部分，直接阻塞到 shared state = ready
future.wait();

// 返回值为future_status
auto fs = future.wait_for() / wait_until();

// 是否还引用着一个shared state
future.valid();
```

## promise从未set_xxx()

也没被析构，则future.get()永远阻塞，因为get会一直阻塞直到state == ready。
如果被析构，则会自动set_exception使得state == ready唤醒其他等待线程并在get时抛出异常
