# 表达式类别：左值，右值

# RVO / NRVO

返回值直接构造在调用栈，而非当前函数栈

RVO是匿名对象，在C++17开始强制RVO。

NRVO是具名对象，已在当前函数栈内尝试构造过的对象，会直接构造到调用栈上。
标准不强制，但几乎所有都实现
``` C++
struct T {
    T() = default;
    T(const T&) = delete;
    T(T&&) = delete;
};

T f() {
    T x;
    return x;   // 只要 NRVO 生效，这就可以通过编译。debug或者-O0会关闭NRVO导致失败
}
```

# unique_ptr

本质是个小对象，里面只保存原始的裸指针 T* 和一个deleter D，析构时直接if (ptr != nullptr) deleter(ptr);

由于独占所有权，因此拷贝构造/拷贝赋值 = delete

deleter决定了不光可以管理new/delete，还可以管理资源文件如FILE* 和socket_fd等

常用方法：
- release():交出指针，自己变空，不析构
- reset(p):析构当前对象，再接管 p
- swap:交换两个的对象指针和deleter指针

# shared_ptr

内部主要有两个指针，一个指向被管理对象T*，另一个指向控制块control_block*(共享)。shared_ptr共享的是控制块，而不是对象本身

控制块里一般存放的是：
control_block
 ├── atomic<long> strong_count  // shared_ptr 数量
 ├── atomic<long> weak_count    // weak_ptr 数量 + **strong_count**
 ├── deleter                    // 删除策略
 └── allocator (可选)

strong_count == 0 对象被析构，但控制块不一定释放，要等weak_count == 0才会析构控制块

不建议用裸指针构造，因为可能构造多个shared_ptr造成最后双重释放，且先分配T再分配control_block，两次分配。

建议make_shared，一次分配完毕。但weak_ptr没释放完毕时，整块内存都不会被释放

引用计数是atomic类型，因此引用计数是线程安全的，但针对对象的操作不是线程安全的。

常用方法：
- reset(p):放弃当前对象，再接管 p，也可以不传参，直接放弃管理，使得强引用数量--
- swap:交换两个的对象指针和控制块指针
- unique:返回当前状态强引用计数是否为1

循环引用需要依靠weak_ptr解决

## enable_share_from_this

直接return this指针会生成一个新的控制块，最后本对象会被析构两次导致崩溃。

shared_ptr 构造过程中，检测到 Foo 继承了 enable_shared_from_this<Foo>， 就把“自己的控制块”写进 Foo::weak_this`。

``` C++
struct Foo : enable_shared_from_this<Foo> {
  shared_ptr<Foo> get_self() {
    return weak_from_this();
    return shared_from_this();
  }
};
```

对象必须已经由 shared_ptr 管理，否则UB，因此不可在构造函数里调用，因为构造期间，shared_ptr还没完成控制块绑定，weak_this还没被设置

# weak_ptr

与shared_ptr类似，也有control_block*,有没有T*看实现，标准不规定。

weak_ptr不能凭空存在，因为要指向控制块，因此构造里要么来自shared_ptr，要么来自另一个weak_ptr

一般通过if (auto sp = wp.lock())获得一个强引用 sp 再进行操作

常用方法：
- lock:尝试升级为shared_ptr，如果对象已析构，返回空shared_ptr
- reset():不再观察
- swap:交换内部control_block的绑定
- use_count:跟shared_ptr一样返回strong_count
