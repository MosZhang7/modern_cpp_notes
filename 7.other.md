# 错误处理有几种方法，为什么有些场合要禁用exception？使用过noexcept关键字吗？它有什么使用功能？

exception的栈展开时间不确定，所以实时系统/嵌入式不能用。且C接口不能throw，要兼容时也不能用。

noexcept表示不会再有异常抛出，如果有，直接崩溃。析构函数默认是noexcept，因为已经在栈展开，再throw直接崩

noexcept可以影响STL，大部分时候STL的决策会使用noexcept的move以增加性能，否则可能会转换成拷贝
同时可以让编译器知道不会生成异常传播路径，优化可以更激进

栈展开过程：
1. 异常抛出：当throw语句执行时，当前函数停止执行，开始栈展开过程
2. 局部对象析构：从当前函数开始，按照创建的反序析构所有局部对象
3. 查找catch块：沿着调用栈向上查找匹配的catch块
4. 匹配处理：找到匹配的catch块后执行处理代码
5. 恢复执行：处理完成后继续执行catch块之后的代码

# final为什么能提高代码执行效率？

只在有虚函数和动态分派的时候可能优化，被final修饰后，可能可以直接调用函数而不用走虚函数表，但性能有限，因此final允许优化，而不是保证优化

# 编译器优化级别

-O0：几乎不做优化，编译最快，一般用于调试
-Og：保持性能但又不影响调试
-O1：启用一批"性价比高"的优化，主要用于嵌入式等边缘设备
-O2：release的默认选择，启用大量成熟优化：积极内联，循环优化，表达式消除，跨基本块优化
-O3：更激进的优化，但不一定更快
-Os：优化体积
-Oz：主要在clang，更极端地优化体积
-Ofast：只要快，不一定要准，主要影响浮点相关

# 怎么分析代码，找出性能优化点？

1. 明确慢在哪里（不是猜）
2. 建立可重复的 benchmark
3. 采样（profile）找热点
4. 判断热点类型（CPU / 内存 / IO / 锁）
5. 有针对性地改
6. 回到 benchmark 验证
